`nuxt-backend` bundles a set of [Convex components](https://www.convex.dev/components), wired into Better Auth and activated by env vars. Each **degrades gracefully** when unconfigured ŌĆö billing queries return `null`, email is skipped with a log, and so on.

| Capability | Component | Entrypoint | Setup helper |
|---|---|---|---|
| Email | [`@convex-dev/resend`](https://www.convex.dev/components/resend) | nested in `backend` | _automatic_ ŌĆö `components.backend.email.send` |
| Billing | [`@convex-dev/polar`](https://www.convex.dev/components/polar) | `nuxt-backend/convex/billing` | `setupBilling` |
| Rate limiting | [`@convex-dev/rate-limiter`](https://www.convex.dev/components/rate-limiter) | `nuxt-backend/convex/rate-limit` | `setupRateLimiter` |
| Migrations | [`@convex-dev/migrations`](https://www.convex.dev/components/migrations) | `nuxt-backend/convex/migrations` | `setupMigrations` |
| Aggregates | [`@convex-dev/aggregate`](https://www.convex.dev/components/aggregate) | `nuxt-backend/convex/aggregate` | `TableAggregate`, `withTriggers` |
| Workflows | [`@convex-dev/workflow`](https://www.convex.dev/components/workflow) | `nuxt-backend/convex/workflows` | `setupWorkflows` |
| Search | Convex native full-text search | `nuxt-backend/convex/search` | `search`, `defineSearch` |

## One call mounts everything

`defineBackendApp(components)` (from `nuxt-backend/convex/app`) is the whole `convex.config.ts`. It mounts every bundled component, declares their typed env vars, and forwards the email config to the nested Resend component ŌĆö so you set `RESEND_API_KEY` and auth OTP / verification / reset email just works.

```ts [backend/convex.config.ts]
import { defineBackendApp } from 'nuxt-backend/convex/app'
import backend from 'nuxt-backend/convex/component/convex.config'
import aggregate from '@convex-dev/aggregate/convex.config'
import migrations from '@convex-dev/migrations/convex.config'
import polar from '@convex-dev/polar/convex.config'
import rateLimiter from '@convex-dev/rate-limiter/convex.config'
import workflow from '@convex-dev/workflow/convex.config'

export default defineBackendApp({ backend, aggregate, migrations, polar, rateLimiter, workflow })
```

You still `import` each component definition here ŌĆö Convex builds its component tree from those imports ŌĆö and hand them in; the wiring is done for you.

::tip
Need extra env vars or your own components? `defineBackendApp` returns the app:

```ts
const app = defineBackendApp(components, {
  env: { STRIPE_SECRET_KEY: v.optional(v.string()) },
})
app.use(myComponent)
export default app
```

For total control, build the app yourself and call `installBackend(app, components)` ŌĆö both from `nuxt-backend/convex/app`.
::

## Why email nests and the others don't

Email is the one component nested **inside** `backend` (it's self-contained, so `backend` exposes it as `email.send`). The other five mount at the **app level** ŌĆö they can't be hidden inside `backend` because a nested component is sandboxed from the parent app, and these operate over **your** tables, functions, auth, and HTTP routes.

Because [components are isolated from the app's env](https://docs.convex.dev/components), the scaffolded `convex.config.ts` declares the deployment env vars with typed validators and **forwards** the email config to the nested component:

```ts [backend/convex.config.ts]
import { defineApp } from 'convex/server'
import { v } from 'convex/values'
import backend from 'nuxt-backend/convex/component/convex.config'
// ŌĆ”and the other five components

const app = defineApp({
  env: {
    RESEND_API_KEY: v.optional(v.string()),
    POLAR_SERVER: v.optional(v.union(v.literal('sandbox'), v.literal('production'))),
    // ŌĆ”see the env-var reference
  },
})
// Resend is nested in `backend`; forward its config by reference.
app.use(backend, {
  env: {
    RESEND_API_KEY: app.env.RESEND_API_KEY,
    RESEND_FROM: app.env.RESEND_FROM,
    RESEND_TEST_MODE: app.env.RESEND_TEST_MODE,
  },
})
// ŌĆ”app.use() the rest
export default app
```

## Server helpers vs. composables ŌĆö two halves, not duplicates

The `nuxt-backend/convex/*` entrypoints are **server** code: you import them *inside your Convex functions* to construct and configure each component client (with sensible defaults baked in ŌĆö auth rate limits, workflow retries, etc.). The auto-imported `useSearch` / `useCount` / `useBilling` / `useWorkflowStatus` composables are **client** code: thin Vue-reactive wrappers over `useQuery` that subscribe to the queries those functions expose.

You need both ŌĆö the helper builds the server endpoint; the composable consumes it reactively in your components.

::note
Each helper lives on its **own** entrypoint (rather than one barrel) so a Convex function only bundles the component it actually imports ŌĆö the same reason the underlying `@convex-dev/*` packages are split, and what keeps `better-auth` out of a file that only does rate limiting.
::
