`setupBilling(components.polar, components.backend, config)` wires subscriptions, discounts, and **prepaid credits** via the [Polar](https://www.convex.dev/components/polar) component, linked to your auth users. The reactive **feature + credit cache** lives inside the bundled `backend` component, so **you add nothing to your schema**.

## Server setup

The config mirrors the Polar client (`getUserInfo`, `products?`, `organizationToken?`, `server?`) plus a query-safe `currentUserId(ctx)` resolver used by the reactive reads.

```ts [backend/billing.ts]
import { setupBilling } from 'nuxt-backend/convex/billing'
import { components } from './_generated/api'

export const {
  api,
  functions,
  webhookEvents,
  spendCredits,
  getCustomerState,
  createDiscount,
  polar,
} = setupBilling(components.polar, components.backend, {
  getUserInfo: async (ctx) => { /* … */ },
  currentUserId: async ctx => /* the signed-in user's id */,
})
```

`setupBilling(...)` returns:

::field-group
  :::field{name="api"}
  `polar.api()` — checkout / portal / product / subscription functions to re-export.
  :::

  :::field{name="functions"}
  Ready-made reactive functions to re-export so the composables work with zero wiring: `getCurrentSubscription` (→ `useBilling`), `getFeatures` (→ `useFeatures`), `getCredits` (→ `useCredits`), and a `syncEntitlements` action to refresh the cache after checkout/top-up.
  :::

  :::field{name="webhookEvents"}
  Typed Polar webhook handlers for `polar.registerRoutes(http, { events })` that keep the cache fresh (subscriptions, benefit grants, credit balances).
  :::

  :::field{name="spendCredits(ctx, { userId, name, meterId?, value? })"}
  Spend prepaid credits from your **own** server code. With `meterId` set it throws when the balance is too low, so credits are never billed as overage.
  :::

  :::field{name="getCustomerState / createDiscount / polar"}
  Live customer state, coupon creation, and the raw client.
  :::
::

## Prepaid credits

Credits are Polar's official model: a credit pack is a one-time product whose *Credits* benefit tops up a meter balance (no backend crediting code needed). The client tops up via `useCredits().topUp(packId)` (a checkout) and the balance is drawn down by `spendCredits`.

## Reactive composables

The auto-imported composables read the cache reactively:

```vue
<script setup lang="ts">
// Subscription state + checkout/portal
const billing = useBilling()
// billing.subscription, billing.isSubscribed, billing.checkout(...)

// Feature-gating
const { has, hasPlan } = useFeatures()
// has('priority_support'), hasPlan('prod_pro')

// Prepaid credits
const credits = useCredits()
// credits.balance, credits.topUp(packId), credits.refresh()
</script>
```

There is also a `<CheckoutLink>` component for launching checkouts.

## Feature-gating by a friendly name

`useFeatures().has(feature)` matches a granted benefit by its Polar benefit id, grant id, `type`, **or any value in the benefit's metadata**. So set a stable key on the Polar benefit (e.g. metadata `{ key: 'premium' }`) and gate with `has('premium')` instead of hardcoding a UUID.

::tip
The cache reads the benefit's metadata **live** (not the grant-time snapshot), so changing it takes effect on the next sync — no re-granting.
::

## Environment

| Variable | Effect when unset |
|---|---|
| `POLAR_ORGANIZATION_TOKEN` | Checkout/portal disabled; subscription queries return `null` |
| `POLAR_WEBHOOK_SECRET` | Polar webhook events aren't verified |
| `POLAR_SERVER` | Defaults to `sandbox` |
