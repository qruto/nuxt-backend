The `nuxt-backend/convex` entrypoint runs in your **Convex backend** (`backend/`), not the Vue app. It exposes the same two client-code patterns described in the Convex component authoring docs — a simple wrapper via `createAuth(ctx, component, options?)` and a ready-made API remount via `makeAuthApi(component, query)` — plus `setupAuth(...)`, a convenience helper that composes both for the common scaffolded case.

## The scaffolded files

If you let the module scaffold the files, you can keep them as generated. If you create them manually, they look like this:

```ts [backend/convex.config.ts]
import { defineBackendApp } from 'nuxt-backend/convex/app'
import backend from 'nuxt-backend/convex/component/convex.config'
import aggregate from '@convex-dev/aggregate/convex.config'
import migrations from '@convex-dev/migrations/convex.config'
import polar from '@convex-dev/polar/convex.config'
import rateLimiter from '@convex-dev/rate-limiter/convex.config'
import workflow from '@convex-dev/workflow/convex.config'

export default defineBackendApp({ backend, aggregate, migrations, polar, rateLimiter, workflow })
```

```ts [backend/auth.config.ts]
export { default } from 'nuxt-backend/convex/auth.config'
```

```ts [backend/auth.ts]
import { setupAuth } from 'nuxt-backend/convex'
import { components } from './_generated/api'
import { query } from './_generated/server'

export const {
  authComponent,
  createAuthOptions,
  options,
  createAuth,
  getAuthUser,
} = setupAuth(components.backend, query)
```

```ts [backend/http.ts]
import { httpRouter } from 'convex/server'
import { authComponent, createAuth } from './auth'

const http = httpRouter()
authComponent.registerRoutes(http, createAuth)

export default http
```

That setup mounts the packaged Better Auth component, configures Convex to trust its tokens, and exposes `api.auth.getAuthUser` for app queries. The scaffolded `backend/auth.ts` re-exports `authComponent`, `createAuthOptions`, `options`, `createAuth`, and `getAuthUser` for app Convex functions to import.

## `setupAuth(...)`

```ts
function setupAuth<DM, Schema>(
  componentRef: ComponentRef,
  queryBuilder: QueryBuilder<DM, 'public'>,
  options?: SetupAuthOptions<Schema>,
): {
  authComponent: AuthComponent
  createAuthOptions: (ctx) => BetterAuthOptions
  options: BetterAuthOptions
  createAuth: (ctx) => BetterAuthInstance
  getAuthUser: RegisteredQuery
}
```

The lower-level pieces — `createAuth`, `createAuthOptions`, `makeAuthApi`, `createBetterAuth`, `createBetterAuthOptions` — are also exported from `nuxt-backend/convex` for when you want to compose them yourself.

## Cross-component wiring

OTP / verification / reset emails are routed through the nested Resend component **automatically** — you don't pass an `email` integration. `integrations` accepts `{ rateLimiter, onUserCreated, email? }`:

- `rateLimiter` throttles OTP sends
- `onUserCreated(ctx, user)` runs on signup (the ctx is a full mutation/action ctx, so it can start a Workflow)
- `email?` is an optional override to bypass the nested transport

```ts [backend/auth.ts]
export const { authComponent, createAuth, getAuthUser } = setupAuth(components.backend, query, {
  integrations: {
    // email is automatic via the nested Resend component (set RESEND_API_KEY)
    rateLimiter,
    onUserCreated: async (ctx, user) => {
      await workflow.start(ctx, internal.workflows.onSignup, {
        userId: user.id, email: user.email, name: user.name,
      })
    },
  },
})
```

Billing links Polar customers to auth users, and its reactive feature/credit cache lives **inside** the `backend` component — so subscriptions, discounts, feature-gating, and prepaid credits work out of the box with nothing added to your schema.

::note
With no `RESEND_API_KEY` configured, OTP codes log to the console instead of sending (graceful no-op).
::
